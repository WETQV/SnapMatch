# Использование SnapMatch

## Первый запуск

1. Запустите `python main.py` или собранный `SnapMatch.exe`.
2. Откройте вкладку настроек.
3. Укажите Telegram token, полученный в BotFather.
4. Добавьте минимум одну AI-модель.
5. Сохраните настройки и запустите бота.

Telegram token имеет вид `123456789:AA...`. Он относится к конкретному боту и хранится локально в конфигурации приложения.

## Модели

Модель в SnapMatch описывает endpoint, ключ доступа и параметры генерации.

Основные поля:

- `id` - имя модели, например `gpt-4.1-mini`, `openai/gpt-4.1-mini` или локальный id;
- `api_key` - ключ провайдера;
- `base_url` - URL API, если используется не стандартный endpoint;
- `api_type` - тип провайдера;
- `supports_vision` - поддержка изображений;
- `context_window_size` - размер контекста;
- `max_concurrent_requests` - лимит одновременных запросов.

Пример OpenRouter:

```txt
id: openai/gpt-4.1-mini
api_type: openai
base_url: https://openrouter.ai/api/v1
api_key: YOUR_OPENROUTER_KEY
```

Пример локального OpenAI-compatible сервера:

```txt
id: local-model
api_type: openai
base_url: http://127.0.0.1:1234/v1
api_key: local-key
```

Если настроено несколько моделей, SnapMatch может выбирать их по очереди, по весу или по наименьшему числу использований. После изменения списка моделей сохраните настройки и перезапустите бота.

## MCP

MCP подключает к модели внешние инструменты: локальные скрипты, файловые операции, браузерные действия, интеграции и другие серверы, совместимые с Model Context Protocol.

Поток работы:

1. MCP-сервер добавляется во вкладке MCP.
2. SnapMatch запускает сервер как отдельный процесс или подключается к удалённому endpoint.
3. Приложение получает список доступных tools/resources/prompts.
4. Модель может запросить вызов tool.
5. SnapMatch проверяет правила доступа и выполняет вызов.

Основные поля MCP-сервера:

- `name` - имя сервера;
- `transport` - `stdio`, `sse` или `streamable_http`;
- `command` - команда запуска для `stdio`;
- `args` - аргументы команды;
- `cwd` - рабочая папка;
- `url` - endpoint для сетевых transport;
- `env` - переменные окружения;
- `access` - разрешённые контексты: admin, private, group, secretary;
- `enabled` - включён ли сервер.

Пример локального MCP-сервера:

```txt
name: simple_snapmatch_mcp
transport: stdio
command: python
args: examples/mcp/simple_snapmatch_mcp.py
cwd: .
enabled: true
```

Способы добавления MCP:

- через вкладку MCP в GUI;
- через пример из `examples/mcp`;
- вручную в `config.json`, если приложение закрыто.

Для MCP лучше использовать только понятные и проверенные серверы. Сервер может получать данные из контекста запроса и выполнять действия в пределах выданных ему прав, поэтому рабочую папку и переменные окружения стоит задавать осознанно.

## Режим секретаря

Секретарь помогает обрабатывать входящие сообщения от имени владельца бота. Режим полезен для личного помощника, клиентских обращений, предварительных черновиков и сценариев, где ответ нужно подготовить перед отправкой.

Как работает поток:

1. Человек пишет боту.
2. SnapMatch сохраняет событие и контекст.
3. Секретарь ждёт короткую паузу, чтобы собрать серию сообщений.
4. Модель формирует черновик, подтверждаемый ответ или автоматический ответ.
5. Ответ отправляется в зависимости от выбранного режима.

Основные режимы:

- `off` - секретарь выключен;
- `draft` - формируется черновик;
- `confirm` - ответ требует подтверждения;
- `auto` - ответ отправляется автоматически.

Настройки секретаря:

- владелец, для которого включён режим;
- режим ответа;
- системный prompt секретаря;
- задержка ответа;
- окно объединения сообщений;
- лимит сообщений в пачке;
- TTL сессии;
- обработка голосовых и изображений;
- правила для отдельных чатов.

Пример сценария:

```txt
Задача: бот принимает вопросы от клиентов.
Режим: confirm.
Пауза группировки: 10-20 секунд.
Ответ: сначала черновик владельцу, затем отправка после подтверждения.
```

Для публичных чатов автоматические ответы стоит включать только после проверки на закрытом тестовом чате.

## Голосовые сообщения

SnapMatch может обрабатывать голосовые сообщения:

- локально через Vosk;
- через OpenAI Whisper;
- через Groq Whisper-compatible модель.

Для локального Vosk используется модель распознавания речи. По умолчанию ожидаемый путь:

```txt
assets/models/stt/vosk
```

Папка с Vosk-моделью не входит в репозиторий, потому что такие модели занимают много места и выбираются под нужный язык. Скачайте подходящую модель Vosk отдельно, распакуйте её в этот путь или укажите свой путь во вкладке голосовых настроек.

## Дополнительные настройки

В дополнительных настройках находятся параметры, которые влияют на поведение приложения в целом:

- длина контекста;
- очистка истории;
- хранение статистики;
- лимиты аудита MCP;
- уведомления;
- reasoning-режимы;
- скрытие внутренних рассуждений модели.

После изменения низкоуровневых параметров лучше перезапустить приложение, чтобы новые значения применились ко всем runtime-компонентам.

## Локальные данные

SnapMatch создаёт локальные рабочие файлы:

- `config.json` - настройки приложения;
- `database.db` - история, пользователи, события MCP и секретаря;
- `.encryption_key` - ключ для локального шифрования секретов;
- `assets/uploads/` - пользовательские вложения;
- `assets/models/` - локальные модели, например Vosk.

Эти файлы относятся к конкретной установке приложения. При публикации форка или своей сборки обычно распространяют исходный код и инструкции, а не личные токены, базу сообщений, загрузки и локальные модели.

## Безопасная настройка

- Telegram token и API-ключи используются только для вашей установки.
- MCP-серверы получают ровно те возможности, которые вы им разрешаете.
- Для файловых MCP-серверов указывайте рабочую папку с минимально нужным доступом.
- Автоответы секретаря сначала проверяйте на тестовом чате.
- Перед публичным релизом убедитесь, что в сборку не попали личные данные, база сообщений и пользовательские вложения.
